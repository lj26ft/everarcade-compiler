# World Factory MVP Phase 3: Generated World Deployment and Remote Verification

Phase 3 extends the World Factory proof from local generation and deterministic replay into a deployable, remotely reachable runtime surface. It demonstrates that the generated Frontier Settlement world can leave the local build path, be wrapped by runtime HTTP endpoints, and be independently verified by a remote client.

```text
Prompt
↓
Blueprint
↓
Contract Plan
↓
Generated world.evr
↓
Execution
↓
Deployment
↓
Remote Verification
```

## Boundary of the Proof

This phase proves generated-world deployment readiness for a single local runtime service. It does **not** claim production federation, multi-operator consensus, global MMO scale, or EverNode provisioning. Those remain future phases.

## Source World

Phase 3 uses the existing generated world package:

```text
examples/world-factory/frontier-settlement/out/world.evr
```

No new template is created. The deployment command copies that generated package into the deployment bundle.

## End-to-End Flow

From the repository root:

```bash
node creator-sdk/cli/everarcade.mjs world factory generate \
  --project examples/world-factory/frontier-settlement

node creator-sdk/cli/everarcade.mjs world factory boot \
  --project examples/world-factory/frontier-settlement

node creator-sdk/cli/everarcade.mjs world factory run --ticks 100 \
  --project examples/world-factory/frontier-settlement

node creator-sdk/cli/everarcade.mjs world factory replay \
  --project examples/world-factory/frontier-settlement

node creator-sdk/cli/everarcade.mjs world factory deploy \
  --project examples/world-factory/frontier-settlement

node creator-sdk/cli/everarcade.mjs world factory serve \
  --project examples/world-factory/frontier-settlement

node creator-sdk/cli/everarcade.mjs world factory remote-verify \
  --project examples/world-factory/frontier-settlement \
  --url http://localhost:8787
```

The short product form is:

```bash
everarcade world factory generate
everarcade world factory boot
everarcade world factory run --ticks 100
everarcade world factory serve
everarcade world factory remote-verify
```

## Deployment Bundle

`world factory deploy` prepares:

```text
examples/world-factory/frontier-settlement/out/deploy/
├── world.evr/
├── runtime-config.json
└── deployment-report.json
```

The bundle records the generated package, local HTTP endpoint profile, package verification status, replay verification status, latest world hash, continuity root, and remote verification status.

## Runtime HTTP Wrapper

`world factory serve` exposes the existing deterministic runtime output through a small local HTTP wrapper. It follows the same proof shape as the Arena Vanguard / HotPocket runtime wrappers: expose health, current state commitment, canonical journal, and verification report without changing the generated world package.

Endpoints:

| Endpoint | Purpose |
| --- | --- |
| `/health` | Reports that the runtime wrapper is running. |
| `/state` | Returns the current world id, tick, state root, world hash, and verification status. |
| `/journal` | Returns the canonical journal generated by runtime execution. |
| `/verify` | Returns package and replay verification status plus commitments and replay material needed by the remote verifier. |

## Remote Verification

`world factory remote-verify` fetches `/state`, `/journal`, and `/verify` from the runtime wrapper. It recomputes commitments locally using the same canonical commitment function as the runtime path and confirms that:

- package verification is `PASS`;
- replay verification is `PASS`;
- the remote state root matches recomputation;
- the remote world hash matches recomputation;
- the remote continuity root matches recomputation.

The verifier prints:

```text
PASS
```

only when the server's claims match independently recomputed commitments.
