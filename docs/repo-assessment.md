## Build and Test Reality Check

### Command outcomes
- `cargo fmt --all --check` → **failed** (format drift in `src-bin-everarcade/src/main.rs`).
- `cargo test --workspace` → **failed initially** due missing `vendor/`.
- `cargo test -p execution-core` → **failed initially** due missing `vendor/`.
- `cargo test -p everarcade-cli` → **failed initially** due missing `vendor/`.
- `bash scripts/vendor_deps.sh` → **passed**, generated `vendor/`.
- `bash scripts/build_runtime_release.sh` → **passed**, built dist tarball.
- `bash scripts/validate_clean_vm_bootstrap.sh` → **passed with warning** (`kill: No such process` after short-lived runtime).

### Root cause analysis for vendor failure
- command: cargo test / cargo fmt check path
- failure: cannot read vendored registry directory `/workspace/everarcade-compiler/vendor`.
- root cause: repo enforces offline vendored registry in `.cargo/config.toml` but `vendor/` absent until bootstrap script is run.
- is it repo bug or environment issue: **repo bootstrap ordering/documentation bug**.
- required fix: make `scripts/vendor_deps.sh` explicit prerequisite in root README + add preflight check in core scripts/tests.

## Architecture Boundary Findings

- `execution-core/src/dag_loader.rs` and `execution-core/src/runner.rs` use filesystem reads directly.
  - finding: domain-core package includes IO helpers, violating strict “pure domain logic” boundary.
  - action: move IO/loading into host/CLI adapter crates and keep `execution-core` record-returning.

- Host orchestration is concentrated in `everarcade-host/src/main.rs` with broad responsibilities (config parse, fs, orchestration, replay, runtime commands).
  - finding: architecture intent is mostly preserved (runtime side owns state/IO), but file size and command branching hinder maintainability.
  - action: split main command handlers into dedicated modules.

- Nondeterminism risk in release script:
  - `scripts/build_runtime_release.sh` uses `date -u` unless `SOURCE_DATE_EPOCH` is provided.
  - action: enforce required deterministic timestamp for CI/release mode.

## Release Readiness Summary
- Real strengths: broad execution-core test surface, release tarball builder, runtime bootstrap validator.
- Critical blockers for v0.1.x readiness:
  1. Missing root README and first-run path.
  2. Offline vendor lock without guaranteed vendor presence.
  3. Domain boundary leaks (fs helpers in `execution-core`).
  4. Repo sprawl and duplicate docs without canonical map.
