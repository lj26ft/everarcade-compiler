# EverArcade v0.1 Critical Path

Canonical assessment date: 2026-06-03

## Release target definition

The shortest credible v0.1 target is **a deployable deterministic world runtime** that can be installed on an operator host or Evernode lease, run continuously, persist state, checkpoint, restart, restore, and prove replay/checkpoint equivalence.

The release should not promise production federation, live XRPL/Xahau settlement, commercial marketplace operation, or production renderer/history infrastructure unless those areas receive new evidence.

## Absolute release blockers

### 1. Stateful runtime host

**Gap:** Deterministic world/runtime primitives exist, but a single production path that loads a package, runs the world loop, accepts inputs, persists replay/journal/checkpoints, exposes health, and shuts down cleanly is not yet proven.

**Definition of done:**

- One command starts a world runtime from a package.
- Runtime emits receipts, journal entries, replay frames/windows, and checkpoints.
- Runtime exposes health/status without affecting consensus authority.
- Runtime handles graceful shutdown and forced restart.

### 2. Durable persistence and restart recovery

**Gap:** Filesystem persistence exists, but production durability needs crash-consistency, locking, fsync, schema versioning, corruption detection, backup, and restore rehearsal.

**Definition of done:**

- Persistence writes survive process crash and host restart.
- Checkpoint and journal corruption are detected with actionable errors.
- Restore from latest valid checkpoint is automated.
- Restore rehearsal is part of release/deployment validation.

### 3. Replay/checkpoint equivalence gate

**Gap:** Replay/checkpoint tests exist, but the deployment gate must prove equivalence from release artifacts and persisted runtime state.

**Definition of done:**

- A generated package can run N ticks, checkpoint, restart, replay, and verify identical state root.
- Gate produces a machine-readable report with package hash, replay root, checkpoint root, and result.
- Gate fails on corrupted replay, missing receipt, broken journal, or mismatched state root.

### 4. Deployment automation from clean host

**Gap:** Many scripts exist, but external operators need one canonical path.

**Definition of done:**

- Clean VM or container path installs dependencies, verifies release artifact, starts runtime, runs smoke world, verifies replay/checkpoint, stops runtime, and uninstalls/rolls back.
- No source-tree-only assumptions except documented development mode.
- Service manager integration is validated.

### 5. Evernode deployment validation

**Gap:** Evernode manifests and local package gates exist, but live lease behavior is unproven.

**Definition of done:**

- Live Evernode lease trial deploys the v0.1 runtime package.
- Runtime persists state on lease storage.
- Kill/restart/recover is demonstrated.
- Resource constraints and disk pressure are measured.
- Operations commands are real provider-backed actions, not simulation strings.

### 6. Operator recovery workflow

**Gap:** Runbooks exist, but recovery is not tied to status, alerts, and gates.

**Definition of done:**

- Operator can identify failed runtime, inspect latest valid checkpoint, restore, verify replay equivalence, and rejoin service.
- Recovery produces logs and status artifacts.
- Recovery drill is run before release.

### 7. Public scope labeling

**Gap:** CLI/docs include many advanced domains that are scaffold-level and can imply production readiness.

**Definition of done:**

- CLI help and docs label renderer/history/federation/XRPL/Evernode live settlement maturity.
- v0.1 quickstart avoids unsupported commands.
- Architecture docs distinguish implemented, experimental, scaffold, and planned domains.

## High priority before v0.1

### Developer experience

- Consolidate to one canonical starter template.
- Add tested `new -> run -> replay verify -> package` workflow.
- Document deterministic constraints and common failure modes.
- Publish SDK facade and mark internal/experimental APIs.

### Runtime observability

- Add health endpoint/status command with package hash, replay cursor, checkpoint age, latest receipt, and recovery state.
- Add structured logs with correlation ids.
- Export metrics outside deterministic authority.
- Link alerts/statuses to runbooks.

### Security and package constraints

- Enforce package size, asset size, memory, fuel, tick, and replay-window quotas.
- Document sandbox boundary and allowed host interactions.
- Add package path traversal/symlink validation.
- Keep live XRPL/Xahau disabled by default.

### Release engineering

- Produce signed release artifact with checksums.
- Validate offline/frozen targeted checks.
- Generate software bill of materials or dependency inventory.
- Ensure generated artifacts are either rebuilt or clearly marked fixtures.

## Post-v0.1

The following should be explicitly outside the v0.1 release promise unless separately implemented and validated:

- Production federation with dynamic topology, split-brain handling, and cross-machine recovery.
- Renderer/projection service, historical replay service, observer archive distribution, and missed-frame client recovery.
- Live XRPL/Xahau settlement, marketplace systems, asset ownership enforcement, and commercial billing.
- GPU market, visual builder, advanced editor, and hosted creator portal.
- Advanced civilization/economy/governance/faction/ecology runtime commercialization.
- Public package marketplace, quarantine, trust/reputation, and moderation workflows.

## Fastest executable plan

| Phase | Outcome | Key gate |
| --- | --- | --- |
| Phase 0 | Scope freeze and label scaffolds | Docs/CLI maturity labels merged |
| Phase 1 | Stateful single-world host | Runtime can tick, persist, checkpoint, and stop |
| Phase 2 | Restart and restore | Forced crash resumes from checkpoint and verifies replay |
| Phase 3 | Clean-host deployment | Fresh VM installs artifact and runs smoke world |
| Phase 4 | Evernode lease trial | Live lease deploy/restart/recover report |
| Phase 5 | Developer preview polish | Tested quickstart and signed release artifact |

## Go/no-go rule

v0.1 may ship as a developer preview only if the single-world runtime, persistence, deployment, and recovery gates pass. It should not ship as commercially ready until operational readiness, supportability, security hardening, and customer lifecycle workflows are implemented.
