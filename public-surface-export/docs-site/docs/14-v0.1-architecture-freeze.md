# 14. v0.1 Architecture Freeze

Freeze date: 2026-06-19

This document refreezes the v0.1 architecture after Tier 1 and Tier 2 certification against the deterministic adversarial linter. It is a scope-control document, not a production-readiness claim.

## Freeze Decision

The v0.1 architecture freeze is approved for a developer-preview architecture centered on a single-world runtime. The runtime authority path is package loading, deterministic execution, state roots, receipts, journals, checkpoints, replay verification, recovery, and operator-visible status.

The freeze does not approve production federation, production renderer infrastructure, production history services, live XRPL/Xahau settlement, commercial marketplace operation, or public production hosting guarantees.

## Tier 1 Certification Boundary

Tier 1 certification is the canonical-document consistency gate. It verifies that the frozen architecture names the release scope, points authority at the runtime path, preserves the release-certification policy, and keeps maturity labels aligned with the public documentation set.

Tier 1 passes only when the architecture freeze document exists, names the required v0.1 scope, keeps the canonical docs newline-stable, and records that renderer/history/federation are scaffold-level runtime domains rather than production authority domains.

## Tier 2 Certification Boundary

Tier 2 certification is the adversarial claim-boundary gate. It searches canonical release and maturity documents for unqualified production claims and validates that the freeze text contains explicit must-not boundaries for advanced domains.

Tier 2 passes only when the documentation does not present v0.1 as production-ready, commercially ready, or live-settlement ready without an explicit negation or future-scope qualifier.

## Frozen v0.1 Architecture Scope

The frozen v0.1 architecture includes:

- a deterministic single-world runtime path;
- package loading and validation;
- deterministic execution and state-root derivation;
- receipts, journals, checkpoints, replay verification, and restore evidence;
- local/operator deployment workflows that can be certified against release artifacts;
- explicit maturity labels for alpha, experimental, scaffold, planned, and reference-only areas.

The frozen v0.1 architecture excludes:

- production federation or split-brain-safe multi-host authority;
- renderer/history/federation as production runtime authority;
- live XRPL/Xahau settlement or wallet-mediated production settlement;
- commercial marketplace, revenue, billing, trust, or moderation guarantees;
- public production testnet or hosted-service commitments.

## Scaffold Domain Rule

Renderer/history/federation remain scaffold-level runtime domains for v0.1. They may provide design notes, deterministic records, local harnesses, or prototype validation evidence, but they must not be described as canonical authority, production multiplayer infrastructure, production client history, or a public runtime guarantee.

## Change Control After Refreeze

After this refreeze, changes that widen v0.1 claims must update this document and rerun the deterministic adversarial linter. Changes that affect execution, state, receipts, checkpointing, recovery, federation, deployment, or release policy must also run the affected targeted validation or certification gate described in `docs/10-release-certification.md`.
